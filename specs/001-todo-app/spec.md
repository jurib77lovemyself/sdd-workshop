# Feature Specification: CLI ToDo Management App

**Feature Branch**: `001-todo-app`  
**Created**: 2026-05-02  
**Status**: Draft  
**Input**: CLI 기반의 ToDo관리 앱을 만들고 싶어요. 사용자: 터미널을 사용하는 개인 개발자. 주요기능: 1. TODo 항목추가(제목 필수, 마감일 선택, 우선순위 선택) 2. 전체 목록 조회(완료/미완료/우선순위로 필터링 가능) 3. 항목 완료 처리(항목 ID로 완료 표시) 4. 항목 삭제(항목 ID로 삭제). 기술스택은 아직 미정.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ToDo 항목 추가 (Priority: P1)

터미널 사용자가 새로운 작업을 앱에 추가합니다. 제목은 필수이며, 마감일과 우선순위는 선택적입니다. 이것이 가장 기본적인 기능이므로 가장 높은 우선순위를 가집니다.

**Why this priority**: 기본 기능으로, 이것 없이는 다른 모든 기능이 의미가 없습니다. 사용자가 먼저 할 수 있어야 하는 작업입니다.

**Independent Test**: 제목만 입력해도 항목이 생성되고, 마감일과 우선순위를 추가 옵션으로 지정할 수 있는지 확인합니다. 단독으로 이 기능만 테스트해도 가치를 제공합니다.

**Acceptance Scenarios**:

1. **Given** 앱이 실행 가능한 상태일 때, **When** 사용자가 제목만 제공하여 항목을 추가하면, **Then** 항목이 생성되고 고유 ID가 반환된다.
2. **Given** 앱이 실행 가능한 상태일 때, **When** 사용자가 제목, 마감일, 우선순위를 모두 제공하면, **Then** 모든 정보가 저장되고 반영된다.
3. **Given** 앱이 실행 가능한 상태일 때, **When** 사용자가 제목 없이 추가하려 하면, **Then** 오류 메시지가 표시되고 항목이 생성되지 않는다.

---

### User Story 2 - 전체 목록 조회 및 필터링 (Priority: P2)

사용자가 저장된 모든 ToDo 항목을 조회하고, 완료 상태, 미완료 상태, 또는 우선순위로 필터링할 수 있습니다.

**Why this priority**: 항목 추가 후 사용자가 자신의 작업 목록을 확인해야 하는 핵심 기능입니다. P2 우선순위는 기본적인 조회 기능이 있으면 사용자에게 가치를 제공하기 때문입니다.

**Independent Test**: 필터 없이 모든 항목을 조회할 수 있고, 각 필터 조건(완료, 미완료, 우선순위별)이 올바르게 작동하는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 여러 ToDo 항목이 저장된 상태일 때, **When** 사용자가 필터 없이 목록 조회하면, **Then** 모든 항목이 표시된다.
2. **Given** 완료된 항목과 미완료 항목이 섞여 있을 때, **When** 사용자가 "미완료" 필터를 적용하면, **Then** 미완료 항목만 표시된다.
3. **Given** 서로 다른 우선순위의 항목들이 있을 때, **When** 사용자가 "높음" 우선순위 필터를 적용하면, **Then** 해당 우선순위 항목만 표시된다.

---

### User Story 3 - 항목 완료 처리 (Priority: P3)

사용자가 특정 항목의 ID를 지정하여 완료로 표시합니다.

**Why this priority**: 사용자 업무 흐름의 핵심이지만, P1과 P2가 먼저 동작해야 이 기능의 가치가 빛납니다.

**Independent Test**: 항목을 완료로 표시한 후 목록 조회에서 상태가 반영되는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 미완료 상태의 항목이 있을 때, **When** 사용자가 해당 항목의 ID로 완료 명령을 실행하면, **Then** 항목의 상태가 완료로 변경된다.
2. **Given** 이미 완료된 항목이 있을 때, **When** 사용자가 다시 완료 명령을 실행하면, **Then** 상태 변경 없이 성공 메시지가 표시된다.

---

### User Story 4 - 항목 삭제 (Priority: P4)

사용자가 특정 항목의 ID를 지정하여 삭제합니다.

**Why this priority**: 사용자 업무 관리에 유용하지만, 핵심 기능(추가, 조회, 완료)이 먼저 동작하는 것이 더 중요합니다.

**Independent Test**: 항목을 삭제한 후 목록에서 제거되었는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 저장된 항목이 있을 때, **When** 사용자가 해당 항목의 ID로 삭제 명령을 실행하면, **Then** 항목이 데이터에서 제거되고 목록에 더 이상 표시되지 않는다.
2. **Given** 존재하지 않는 ID로 삭제를 시도할 때, **When** 사용자가 삭제 명령을 실행하면, **Then** 오류 메시지가 표시되고 아무것도 삭제되지 않는다.

---

### Edge Cases

- 동일한 ID를 가진 항목으로 여러 작업을 시도할 때의 동작
- 마감일이 과거인 항목의 표시 방식
- 매우 긴 제목(예: 1000자 이상)을 입력했을 때의 처리
- 저장된 데이터가 없을 때 조회 요청에 대한 응답

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 제목을 입력하여 새로운 ToDo 항목을 생성할 수 있어야 한다.
- **FR-002**: ToDo 항목 생성 시 마감일(선택)과 우선순위(선택) 정보를 추가할 수 있어야 한다.
- **FR-003**: 시스템은 각 항목에 고유한 ID를 할당하여 관리해야 한다.
- **FR-004**: 사용자는 저장된 모든 ToDo 항목의 목록을 조회할 수 있어야 한다.
- **FR-005**: 사용자는 완료/미완료 상태로 항목을 필터링할 수 있어야 한다.
- **FR-006**: 사용자는 우선순위 기준으로 항목을 필터링할 수 있어야 한다.
- **FR-007**: 사용자는 항목 ID를 지정하여 항목을 완료로 표시할 수 있어야 한다.
- **FR-008**: 사용자는 항목 ID를 지정하여 항목을 삭제할 수 있어야 한다.
- **FR-009**: 시스템은 사용자의 ToDo 데이터를 영속적으로 저장해야 한다.
- **FR-010**: 시스템은 CLI 인터페이스를 통해 모든 기능을 지원해야 한다.

### Key Entities

- **Todo Item**: 사용자의 작업 항목을 나타내며, 다음 속성을 포함한다:
  - `id`: 고유 식별자 (필수)
  - `title`: 작업 제목 (필수)
  - `description`: 작업 상세 설명 (선택)
  - `dueDate`: 마감일 (선택)
  - `priority`: 우선순위 (선택) - "높음", "중간", "낮음" 수준
  - `completed`: 완료 상태 (필수, 기본값: false)
  - `createdAt`: 생성 시각 (필수)
  - `updatedAt`: 마지막 수정 시각 (필수)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 CLI를 통해 새로운 ToDo를 30초 이내에 추가할 수 있다.
- **SC-002**: 시스템이 1000개 이상의 항목을 저장하고 조회할 수 있다.
- **SC-003**: 목록 조회와 필터링이 100개 항목 기준 2초 이내에 완료된다.
- **SC-004**: 사용자가 제목만 제공했을 때와 모든 옵션을 제공했을 때 모두 항목이 정상 생성된다.
- **SC-005**: 잘못된 명령어나 입력에 대해 명확한 오류 메시지가 표시된다.
- **SC-006**: 애플리케이션 재실행 후에도 이전에 저장된 모든 데이터가 복구된다.

## Assumptions

- 사용자는 터미널/명령줄 환경에 익숙한 개인 개발자입니다.
- 사용자는 자신의 개발 환경에서만 이 앱을 사용합니다. (다중 사용자 시나리오 제외)
- 로컬 파일 시스템에 데이터를 저장하는 것으로 충분합니다. (별도 데이터베이스 서버 불필요)
- 동시성 처리(동시 접근)는 고려 대상이 아닙니다.
- 데이터 암호화나 인증은 이 버전의 범위 밖입니다.
- REST API나 GUI는 이 프로젝트의 범위 밖입니다.
- 기술 스택은 이후 Plan 단계에서 결정됩니다.
